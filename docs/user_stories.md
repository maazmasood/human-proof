# User Stories & Success Metrics
 In Practice

This document outlines the desired outcomes for the various stakeholders of the Human-Proof protocol.

## 👥 For End-Users (Privacy & Convenience)

### As a privacy-conscious user...
I want to prove I am human without uploading my ID or scanning my iris into a centralized database, so that I can participate in online communities without being tracked or surveilled.

### As a mobile user...
I want to verify my presence using my phone's FaceID or Fingerprint sensor, so that I don't have to solve frustrating and time-consuming CAPTCHAs.

### As a member of a DAO or online voting platform...
I want to know that my vote has more weight than a bot's, so that the community's decisions are based on real human consensus rather than automated manipulation.

---

## 🛠️ For Developers (Integration & Security)

### As a backend engineer...
I want a simple, middleware-based solution to protect my sensitive API endpoints (voting, account creation), so that I can prevent mass-scale automation without building a complex security infrastructure.

### As a product manager...
I want to adjust the "Humanity Threshold" based on the risk level of the action (e.g., higher trust required for financial rewards than for liking a post), so that I can balance security with user experience.

### As a security researcher...
I want to see verifiable hardware attestation data about the device being used, so that I can build sophisticated trust scoring models to identify "phone farms" and emulator-based attacks.

---

## 🏢 For Communities & Platforms (Trust & Scale)

### As a social media moderator...
I want to ensure that every "Like" and "Follow" is backed by a human presence proof, so that the platform's metrics are meaningful and resistant to engagement farming.

### As a non-profit infrastructure provider...
I want an open-source, permissionless protocol to verify humans, so that I am not dependent on proprietary, centralized, and expensive third-party identity services.
